## Commands

```bash
# Install
pip install -e ".[dev]"          # core + CLI + test/lint toolchain ([dev] pulls [web,tui,crypto])
pip install -e ".[dev,agent]"    # + LLM stack for the agent layer (langchain/langgraph/deepagents)
pip install -e ".[dev,interop]"  # + httpx FactoryClient/worker (tg_content_factory)
# As a library: base = core+CLI only; extras: [web] [tui] [crypto] [agent] [interop] [all].
#   CLI imports them lazily; `serve`/`tui`/`worker` fail with a "pip install 'tg-messenger[web]'" hint.

# Run the full test suite
pytest

# Single file / single test
pytest tests/test_client.py
pytest tests/test_client.py::test_name -x

# Lint (ruff: E, F, I, N, W; line-length 120)
ruff check . && ruff check --fix .

# Run the app
tg-messenger --help          # full command list: login/logout/profiles, dialogs/search/read,
                             #   send/forward/edit/delete/mark-read, listen/chat/watch,
                             #   moderate(-rules), heartbeat, suggest, ghostwrite(-dialogs),
                             #   worker, username, serve, tui, agent
tg-messenger -v <cmd>        # DEBUG logging; file log: ~/.tg_messenger/logs/tg_messenger.log (env TG_LOG_DIR)
tg-messenger serve           # web (FastAPI/uvicorn), default port 8090 (env TG_WEB_PORT; --port overrides)
tg-messenger tui             # Textual TUI
tg-messenger agent           # AI auto-reply: needs the [agent] extra + TG_AGENT_* env (see .env.example)
scripts/e2e/run_safe.sh      # manual real-CLI Telegram smoke checks; NEVER pytest/CI
```

CI (`.github/workflows/ci.yml`): every PR and push to main runs `ruff check .` + `pytest -q` on python 3.11/3.12, plus a second job with the `[agent]` extra so the agent tests run un-skipped.

Runtime requires `TG_API_ID` and `TG_API_HASH` (Telegram API credentials) — from the environment or from a `.env` in the cwd, which the CLI entrypoint auto-loads (`_load_dotenv` in `cli/main.py`; real env wins, see `.env.example`). Unit tests never need them — they patch the network seams (see below).

pytest is configured for `asyncio_mode = auto` (no `@pytest.mark.asyncio` needed), a 30s per-test timeout, and `filterwarnings = ["error"]` — **any warning fails the test**.

## Project-wide invariants

- **No silent failures — project-level rule.** Every caught-and-suppressed exception MUST be logged (`logger.exception`, or `logger.warning` for expected drops). A bare `except: pass` is a review bug. This applies to every module, not just the examples listed under logsetup below.
- Every network call goes through `run_with_flood_wait_retry`; never `get_entity('@username')` in loops (~50 resolves in a row → flood).
- UIs render Pydantic models from core — never raw Telethon objects.
- Tests: fakes from conftest only — no network, no real LLM, no real `sleep` (inject `clock`/`rng`); `filterwarnings=error`.
- Manual real-API smoke checks live in `scripts/e2e/` as human-run shell scripts over the installed CLI; never wire them into `pytest` or CI. Safe mutations are confined to Saved Messages / Избранное and clean up their own artifacts. Dangerous/account-visible checks are documented only as parity stubs in `scripts/e2e/99_dangerous_parity.sh`.
- **Every PR must contain a closing keyword (`Closes #N`)** for its issue; one issue = one closing PR.
- Secrets, session strings, phone numbers and login codes never reach logs or the repository.

## Structure & placement rules

All modules from the umbrella-#6 decomposition (issues #8–#26) are implemented; issue numbers below are provenance for `git log`/issue archaeology. **Placement rules — agents must not improvise locations:**
- Services that sit *above* the client (listen + act loops) live in `core/` next to `watch.py` — NOT inside `client.py`, which stays a thin wrapper.

- HTTP to tg_content_factory lives ONLY in `interop/` (httpx, optional extra); `core/` never imports httpx.

## Architecture

A single UI-agnostic core wrapped by three interchangeable front-ends, plus an optional agent layer and the factory interop. `PLAN.md` holds the full design and the TDD build sequence (cycles 0–38 core/UI/agent/tabs, 39–135 the umbrella-#6 decomposition: cache, packaging, crypto/SSO, profiles, search, storage, events, actions, moderation, suggester, ghostwrite, heartbeat, interop, media, usernames, web-auth, ratelimit, UI-login); the project is built test-first.

```
core/   ← all Telegram logic, no UI imports
cli/    ┐
tui/    ├ thin adapters over core, each independently runnable
web/    ┘
agent/  ← LangGraph intent orchestrator over core (optional [agent] extra); core never imports it
```

### Core (`src/tg_messenger/core/`)
- **`client.py` — `StandaloneTelegramClient`**: the only thing the UIs talk to. Thin async wrapper over one Telethon client exposing `dialogs()`, `history()`, `send_text()`, `send_media()`, `download_media()`/`download_message_media()`, `get_me()`, `entity_title()`, `send_reaction()` (`SendReactionRequest` via retry), and **seven** event streams (all the same lazy-bus + drop-oldest fan-out pattern; an unsubscribed bus is a no-op publish): `listen()` (incoming from private chats only — DMs + bots), `listen_all()` (incoming from EVERY chat, groups/channels included — the UIs' live feed), `listen_outgoing()` (own messages from any device, groups INCLUDED — no DM filter), `listen_deleted()` (`MessagesDeletedEvent`; Telegram names the chat only for channels/supergroups), and the #14 trio `listen_chat_actions()` (`ChatActionEvent` — joins/leaves/kicks/title/pin/photo via `events.ChatAction`; kind from the event's `user_joined`/`user_added`→`join`, `user_kicked`→`kick`, `user_left`→`leave`, `new_title`/`new_pin`/`new_photo`, else `other`; `user`/`actor` best-effort), `listen_reads()` (`MessageReadEvent`; `outbox=True` = the OTHER party read OUR messages up to `max_id`) and `listen_reactions()` (`ReactionEvent` via `events.Raw(UpdateMessageReactions)` — changed standard emoji `emoticon` comes from `recent_reactions`, aggregate `results` are not used; custom/premium reactions and unreadable shapes → `None` with a warning; `actor_id` best-effort `None`). `IncomingEvent.album_id` carries the message's `grouped_id` (v1 only marks albums — no aggregator). Telethon handlers registered eagerly at `connect()` — `_on_new_message` publishes into both incoming buses (`_bus_all` always, `_bus` when `is_private`); each #14 handler maps the raw Telethon event to its Pydantic model in a try/except that logs and keeps the stream alive (reactions `logger.warning`, the rest `logger.exception`). `dialogs(dm_only=True)` (default) returns DMs only (filtered `kind == "dm"`); `dm_only=False` returns everything, each `Dialog` with `kind` (`dm|group|channel|bot`); `group_dialogs()` returns every non-DM dialog (the UIs' "Группы" tab — the `kind != "dm"` filter lives here once, not in each front-end). `Dialog.id` is the **marked** peer id (negative for groups/channels) — the same value events carry in `chat_id`, accepted by `history`/`send_text`. **Every network call routes through `run_with_flood_wait_retry`.** **Read anti-flood (cache.py, single-flight):** every user-triggered read goes through a TTL cache — `dialogs()` is ONE full mapped list (all kinds, `maxsize=1`, TTL 30s) and `dm_only` filters from it (tab switching after the first load = 0 network calls); `history()` is keyed `(int(peer), limit, offset_id)` (TTL 15s, `maxsize=64`). Both return a copy (UIs render, never mutate). The dialogs cache is NOT invalidated by events (an active group would defeat it — that was the incident). Every write and every live event invalidates `history` for its peer: `send_text`/`send_media` after a successful send; `_on_new_message`/`_on_outgoing_message` right after computing `dialog_id`, BEFORE mapping (a broken event still drops stale history); `_on_deleted` by `chat_id` if present, else the whole history cache. `download_message_media` is NOT cached (a point fetch by id). **`send_media(peer, file_path, *, caption=None, voice_note=False, video_note=False, force_document=False)` (#21)** maps the three bool flags straight onto Telethon `send_file` and validates the path BEFORE any network call — a missing path or a directory raises `ValueError("file not found: ...")` (no FloodWait budget burned on a typo), then invalidates the peer history on success. `flood_sleep_threshold=0` in `_default_factory` — Telethon never sleeps silently on a FloodWait, every wait surfaces through `run_with_flood_wait_retry`. **Username resolution is expensive (~50 in a row → flood): never `get_entity('@username')` in loops — the agent allowlist resolves through the dialog list once; `watch` memoises titles itself.**
  **Message actions (#15):** `send_text(peer, text, reply_to=None)` quotes a message; `forward(from_peer, ids, to_peer)`, `edit_text(peer, id, text)`, `delete_messages(peer, ids, revoke=True)` and `mark_read(peer, max_id=None)` are thin retry-wrapped Telethon calls. Every message mutation (send/forward/edit/delete) invalidates the `history` cache of its own peer(s) — `forward` invalidates BOTH the source and destination, and logs/filters partial `None` results so already-forwarded messages are not retried as failures. `mark_read` is NOT a mutation of history, but after a successful ack it invalidates the dialogs cache so unread badges do not stay stale; auto-read in web/TUI passes the max message id from the loaded history snapshot so newer unseen messages stay unread. It is best-effort (failures are logged, never raised — web: tracked background task, never awaited in the messages route; TUI: a `_mark_read` worker after history loads, never awaited in a handler). `Dialog.unread` (from telethon `dialog.unread_count`) and `Message.reply_to_id` (best-effort `raw.reply_to.reply_to_msg_id`) are the supporting model fields; UIs render the unread count as a badge (web `<span class="unread">`, TUI `(N)`).
  Delete safety: before Telethon deletion, `delete_messages` fetches the requested ids from the target peer and rejects missing/mismatched messages; `revoke=False`/CLI `--for-me` is rejected for channels/supergroups because Telegram deletes there for everyone.
  **Moderation primitives (#16):** `mute_user(peer, user_id, until_sec)` and `ban_user(peer, user_id)` are thin retry-wrapped `edit_permissions` calls (mute = `send_messages=False` until a future `until_date`; ban = `view_messages=False`); `moderation_rights(peer)` reads `get_permissions(peer, me)` best-effort and returns granular moderation-capable rights (`delete_messages`, `ban_users`; failures → both false with a warning, never raises); `is_admin(peer)` is the compatibility boolean over those rights.

- **`watch.py` — `DeletionWatcher`** (`tg-messenger watch`): backs up own deleted messages (e.g. removed by group moderator bots) to Saved Messages. A bounded cache of recent own messages (`OrderedDict`, 1000) is the only way to recognise OUR messages among deletions — `deleted_ids` carry no author; matching pops the entry (no duplicate notifications). Events without `chat_id` (DMs/small groups) must never match channel cache entries (`is_channel_or_megagroup_id` from `client.py` — the single marked-id threshold) — per-channel message ids collide with the global id space. Messages in the self-dialog are not cached (no notification loop). `run()` uses `asyncio.gather`, NOT `TaskGroup` — TaskGroup wraps KeyboardInterrupt into BaseExceptionGroup and breaks the CLI Ctrl+C pattern. Best-effort by design: Telegram doesn't always send the deletion update; cache is in-memory.
- **`moderation.py` — `ModerationEngine`** (`tg-messenger moderate`): auto-moderates group chats by rules stored in SQLite (#13). Pydantic `ModerationRule(chat_id, name, enabled, conditions, actions)`; `RuleConditions` (regex `pattern` validated fail-fast, `has_link`, `is_forward`, `from_new_member_within_sec`, `max_messages_per_minute`) match with **AND** semantics via `rule_matches(...)`; `RuleActions(delete, mute_sec, ban, warn_text)`. CRUD (`add_rule`/`list_rules`/`remove_rule`, conditions/actions as JSON, PK `(chat_id, name)` → upsert) + a `moderation_log` journal register through `register_moderation_migrations(storage)`. The engine follows the `watch.py` service pattern: `run()` fans out `listen_all()` + `listen_chat_actions()` via `asyncio.gather` (NOT TaskGroup — Ctrl+C), bounded `OrderedDict` caches track new-member join times and per-sender sliding 60s rate windows, `clock` injected (tests never sleep). **Dry-run is the default** — actions only fire under `enforce=True`; otherwise it logs `would …` and journals `dry_run=1`. First matching enabled rule wins; each action runs independently and a failure is `logger.exception`'d AND journaled as `<action>:failed` while the engine keeps going; every decision is journaled. On start the CLI calls `check_admin_rights`: it inspects stored rule actions and requires `delete_messages` only for delete rules and `ban_users` only for mute/ban rules; warn-only rules need no admin right. Chats without needed rights are disabled with a warning, not fatal. `moderate-rules add|list|remove` manage rules (`add` reads a JSON file — see `moderation.json.example`; negative chat ids need `--` before them). `Message.is_forward` (best-effort from the raw Telethon message) is threaded into `process_message`, so forward-rules do fire.
- **`flood.py`**: dependency-free FloodWait retry. Transient waits (≤60s, within a 120s budget) are slept-and-retried; anything else raises `HandledFloodWaitError`. Other exceptions propagate. Vendored from the parent `tg_content_factory` project — keep it pool/DB-free.
- **`cache.py` — `TTLCache`**: dependency-free TTL + maxsize-eviction (oldest first, `OrderedDict` — pattern from `watch.py`) + async single-flight `get_or_fetch` (per-key `asyncio.Lock`, double-check under the lock, a failing fetch is neither cached nor left holding a lock). `clock` is injected (`time.monotonic` default) — tests advance a fake clock, never sleep. Used by `client.py` for the dialogs/history read caches.
- **`ratelimit.py` — `TokenBucket`**: one global cap on ALL outgoing mutations (`send_text`/`send_media`/`forward`/`send_reaction`) — after the #6 decomposition four subsystems send (agent/heartbeat/moderator/worker), and a systematically high send rate risks an account ban (worse than any FloodWait). `acquire()` **waits, never errors** (a send slows down, nothing is lost) and logs a WARNING; `clock`/`sleep` injected (tests never really sleep); concurrent acquirers form a fair queue (`asyncio.Lock`). `acquire()` runs ONCE before the retry loop (a retry after FloodWait does not burn a new token — accepted compromise). Wired into `client.py` via `send_rate_per_min` (env `TG_SEND_RATE` in `make_client`, 0 = off; burst = one minute's worth). **Reads (dialogs/history) are NOT limited** — the #8 cache covers them.
- **`search.py` — `filter_dialogs(dialogs, query)`**: pure, network-free dialog filtering over an already-fetched `list[Dialog]` (the #8 read cache, so the UIs filter without a request — title substring case-insensitive, username with/without `@` exact-or-prefix, id exact plus the positive form of a marked id; empty query → all). Message search lives on the client: **`search_messages(peer, query, limit=20)`** = `iter_messages(search=query)` through `run_with_flood_wait_retry`, NOT cached (a point lookup, not a re-read page). Global content search across all chats is deliberately absent — that's tg_content_factory's job (umbrella #6).
- **`events.py` — `EventBus`**: asyncio fan-out. One Telethon `NewMessage` handler `publish()`es; each UI `subscribe()`s its own bounded queue. **Publishing never blocks** — a full subscriber queue drops its oldest item so a slow consumer can't stall the Telethon loop.
- **`usernames.py`** (`tg-messenger username suggest/set/clear`, #22): own-account public @username generation + availability. `generate_candidates(base, *, count, rng)` is a PURE, deterministic function — `rng` is INJECTED (tests pass `random.Random(0)`; no network, no global random): it normalises `base` to Telegram's rules (translit Cyrillic via a small dict, drop other non-ASCII, strip leading digits/underscores, pad too-short stems) then layers numeric/year suffixes, underscore joins and moderate leet swaps. Every candidate passes `is_valid_username` (5–32 chars, `[a-z][a-z0-9_]*`, no `__`, no trailing `_`); deduped, order-stable, `count` is a ceiling. `check_username`/`set_username`/`clear_username` live on `client.py` (telethon `account.CheckUsernameRequest`/`UpdateUsernameRequest` through `run_with_flood_wait_retry`; `UsernameInvalidError`/`UsernameOccupiedError` map to `ValueError` with a clear message, never a raw traceback; occupied → `check` returns `False`). `find_available(client, base, *, limit, count, rng)` checks candidates **sequentially** (one `check_username` at a time — the anti-flood discipline, NOT a parallel storm) and stops at the first `limit` free names, so ≤ `count` network checks ever happen.
- **`auth.py`**: `SessionStore` persists Telethon `StringSession` strings as 0600 files under `~/.tg_messenger/sessions/` (no SQLite). An external session string can be injected and is never written to disk. **Optional at-rest encryption** (`session_cipher.py`): with `encryption_key` (CLI/UIs pass env `SESSION_ENCRYPTION_KEY`) sessions are stored as `enc:v2:` Fernet tokens — **byte-compatible with tg_content_factory, so a shared key = SSO**; reading a plaintext file under a key lazily rewrites it encrypted; reading an encrypted file with no key raises with a `SESSION_ENCRYPTION_KEY` hint. `enc:v1:` is read-only legacy. **Session strings never reach logs.** `LoginFlow` is the two-step phone→code→2FA sign-in; `phone_code_hash` stays bound to the same client/session. **`LoginSession`** (#26) is a small state machine (`phone`→`code`→`password`→`done`, `property state`) over `LoginFlow`, shared by the web `/tg-login` wizard and the TUI login screen so both UIs drive the SAME flow: `submit_phone`/`submit_code`/`submit_password`/`resend`. User-correctable failures raise `LoginError` (human text) WITHOUT advancing state — the step is just retried; this covers the wrong code, the wrong 2FA password AND bad-phone errors in `submit_phone` (`PhoneNumberInvalid/Banned/Flood` → `LoginError`, so the web wizard renders them in the form instead of a 500 — when extending, normalise new telethon errors here, not in the UIs); a step called out of order raises `RuntimeError`. The whole flow MUST run over one connected client (the login `code` binds to that single session — never split steps across connections); the phone number and code never reach the logs (tested). `delivery_hint(CodeDelivery)` is the shared "where the code went" text. CLI `login --export-session`/`--import-session` move a session between machines/projects.
- **`session_cipher.py`** — `encrypt_session`/`decrypt_session`/`is_encrypted`: dependency-light Fernet wrapper. `enc:v2:` = Fernet over PBKDF2-HMAC-SHA256(secret, salt=`b"tg_session_key_v2"`, 200k iters, 32 bytes) — the factory's scheme reproduced byte-for-byte (a compat test re-derives the key with the constants hardcoded independently). `cryptography` is the `[crypto]` extra, imported lazily — a missing extra under a configured key raises a `pip install 'tg-messenger[crypto]'` `ValueError`, not an ImportError traceback; plaintext passes through so encryption stays opt-in.
- **`storage.py`** — `Storage`: stdlib `sqlite3` behind `asyncio.to_thread`, serialised by one `asyncio.Lock` over a single `check_same_thread=False` connection (concurrent `gather` callers never race or deadlock; `close()` also takes the lock). WAL + `foreign_keys=ON`. Consumers (#16/#17/#19) `register_migrations([...])`; unapplied migrations are tracked by stable statement ids in `_tg_messenger_migrations` and applied inside one transaction (`PRAGMA user_version` mirrors the applied count for inspection; a failing batch rolls back and metadata doesn't advance). The `kv` table (JSON values) is always present and un-versioned; `get_value`/`set_value` cover small odds and ends. `default_db_path(profile)` = `~/.tg_messenger/<safe-profile>.db` (one DB per profile, #11). **The TTL read cache does NOT live here** — it stays in-memory (#8); `client.py` does not depend on storage (the client stays light; storage is a sibling for services, watch.py pattern).
- **`models.py`**: Pydantic v2 domain models (`Dialog`, `Message`, `User`, `MediaRef`, `IncomingEvent`) shared across all interfaces. UIs render these, never raw Telethon objects — mapping happens in `StandaloneTelegramClient._to_message`.
- **`logsetup.py`** — `setup_logging(verbose=, console=)`: rotating file log (always, INFO+; DEBUG with `-v`) + stderr handler (ERROR+ only, one-line, skips `tg_messenger.cli` records — the CLI speaks via click; tracebacks go to the file only). Every CLI invocation calls it; the `tui` command re-runs it with `console=False` (stderr corrupts the alternate screen). Tests are isolated via the autouse `TG_LOG_DIR` fixture in conftest. **No silent failures**: anything caught-and-suppressed must be logged (`logger.exception`/`warning`) — see `_on_new_message`, `EventBus.publish` drops, the chat listener task, SSE streams and the web `Exception` handler.

### Interfaces
All three offer a DM/groups split over the same core API (`?tab=` / Tabs / `--groups`); the groups view is every non-DM dialog (groups, supergroups, broadcast channels, bots), served by `client.group_dialogs()` — the front-ends never re-filter by kind themselves. Each front-end also exposes dialog id (`{id} — title`) and search via `core.search`: web has a `?q=` filter on `/dialogs` plus a `/dialogs/{id}/search?q=` message search and an `<input name="q">` over the list; CLI has `dialogs --find QUERY` (local filter, composes with `--groups`) and a `search PEER QUERY [--limit]` command; the TUI filters the loaded dialog list locally via an `Input#search` (in-dialog message search in the TUI is a v1 deferral). Dialog filtering is always `filter_dialogs` over the cached list — never a network call.
- **Multilogin (`--profile`, #11)**: `--profile NAME` is a sweeping global option (CLI/TUI/serve) mapped onto `session_name`; a profile is just a saved session file (`SessionStore.list_profiles()`). With no flag and >1 saved profile, the CLI and TUI pop an interactive selection menu, while a non-interactive stdin is a clear error (`pass --profile NAME`); 0 or 1 profile resolves silently. One process = one profile (the active one is fixed at startup). The log file is isolated per non-default profile (`tg_messenger_<profile>.log`, set in `logsetup.setup_logging(profile=)`) so two accounts don't interleave. `tg-messenger profiles` lists them; the web exposes a read-only `GET /profiles` (saved names + the active one flagged), `SessionStore` dir overridable via `TG_SESSION_DIR`. Profile lifecycle: `logout [--profile NAME]` does a best-effort Telegram `log_out()` (a dead/revoked session never blocks the local cleanup) and deletes the session file (`SessionStore.delete`); `profiles remove NAME` deletes ONLY the file, no network — for dead sessions. Both confirm unless `--yes`.
- **`cli/main.py`**: click group. `make_client(**kwargs)` builds the client from env — a thin delegate to `core.client.client_from_env`, the SINGLE env→client plumbing all three front-ends share (web/TUI `_make_real_client` delegate too; new env knobs go there, not per-UI); `session_store_from_env` (`core/auth.py`) is the same single source for the SessionStore. `dialogs --groups` lists non-DM dialogs with a `[kind]` marker. **`send` media flags (#21):** `--file PATH` plus `--caption TEXT` (else the positional TEXT is the caption) and the mutually-exclusive `--voice`/`--video-note`/`--as-file` (→ `voice_note`/`video_note`/`force_document`; ≥2 set → `ClickException`). **Packaging:** web/tui are optional extras — `serve`/`tui` import `tg_messenger.web`/`tg_messenger.tui` lazily inside the command and, on `ImportError`, raise a `ClickException` with a `pip install 'tg-messenger[web]'`/`[tui]` hint (base install must not pull fastapi/textual — locked by `tests/test_packaging.py`, a fresh-subprocess `sys.modules` check). The public API (`tg_messenger.__all__`, `tg_messenger.core.__all__`) is snapshot-pinned in the same test; `src/tg_messenger/py.typed` ships the typing marker. **`chat` REPL** runs TWO printer tasks — `listen()` (`←` incoming DMs) and `listen_outgoing()` (`→` our own messages sent elsewhere, deduped against a bounded `sent_ids` so our own typed line isn't echoed back). **Stream-consumer contract (`tests/test_stream_consumers.py`):** a grep guard (the `test_packaging.py` pattern) asserting every `listen_*` stream on the client has a real consumer outside `client.py`, and that all three live UIs (TUI/web/CLI `chat`) consume `listen_outgoing()`. `listen_reactions` is in the test's `_DEFERRED` set (published but no UI renders reactions yet — tracked in #56); remove it there the moment a consumer lands.
- **`web/app.py`**: FastAPI + server-rendered HTMX fragments + SSE for live messages (`sse_event_stream` **merges `listen_all()` + `listen_outgoing()`** by racing both iterators' `__anext__` — incoming AND our own messages sent from another device, the latter framed with `out:true` so `chat.html` renders them right-aligned `msg out`; group streams work too). Echoes of messages this server itself sent are skipped via `app.state.sent_ids` (bounded `OrderedDict`-as-set, fed by `/send` & `/media` through `_remember_sent`) so the optimistic bubble the POST returned isn't duplicated — the single source of truth, not a client-side guess. `build_app(client=..., session_name=..., web_pass=...)`. The client connects/disconnects in the FastAPI lifespan. `GET /dialogs?tab=dm|groups` (unknown tab falls back to dm); the tab buttons live in `chat.html`. **Web auth (#24):** `web_pass` (from `TG_WEB_PASS`) gates EVERY route behind an HMAC-cookie session via an `@app.middleware("http")` — when it's None NO middleware is added (full backward compat: existing `build_app(client=stub)` tests stay green). Cookie value is `"{expiry}:{hmac_sha256}"` signed with a per-process random key (`secrets.token_bytes`, on `app.state.cookie_key` — restart invalidates all sessions); `_valid_cookie` checks the signature (`hmac.compare_digest`) AND expiry. `/login` (form + POST — the only public paths, with `/logout`) compares the password constant-time over UTF-8 **bytes** — `hmac.compare_digest` raises TypeError on non-ASCII *str*, and `TG_WEB_PASS` may be Cyrillic (see `.env.example`), so never compare the raw strings; wrong password → injectable `_login_delay` (tests monkeypatch to a no-op, NO real sleep) + `logger.warning` with `request.client.host` + 401. Unauthenticated requests redirect browsers (Accept: text/html) to `/login`, else 401 (covers JSON + `/stream/{id}` SSE). The `serve` CLI refuses a non-localhost `--host` without `TG_WEB_PASS` unless `--insecure` (a logged, deliberate bypass); 127.0.0.1 without a password works as before. No built-in HTTPS — TLS is a reverse-proxy concern. `GET /profiles` is a read-only list of saved profiles (the served `session_name` flagged active). **`POST /dialogs/{id}/media` (#21)** is the multipart upload (`UploadFile` + optional `caption` Form): it **streams** the file to a `NamedTemporaryFile` in bounded 1 MiB chunks (`_UPLOAD_CHUNK_BYTES`) — the upload never sits in memory whole, and reading STOPS as soon as the size limit is crossed (over-limit → 413, empty → 400; cap = `TG_WEB_MAX_UPLOAD_MB`, default 50, `_max_upload_mb()` falls back on a bad/non-positive value with a warning), then calls `client.send_media` and **deletes the temp file in a `finally`**; returns the same bubble fragment as `/send`. Never replace the chunked loop with a whole-body `file.read()` — that re-opens the memory-exhaustion hole the limit exists to close. The composer's 📎 button (chat.html) posts the picked file with the current composer text as caption.
- **`tui/app.py`**: Textual app, `MessengerTUI(session_name=...)`. Two non-obvious constraints: `on_mount` resets the loop's task factory **before** the client exists — Textual's `eager_task_factory` (py3.12+) breaks Telethon, see the comment in `on_mount`; don't remove or move that line. And all network work goes through `self.run_worker(...)`, never `await` in handlers — awaiting stalls the message pump. DM/Группы are a `SidebarTabs` (a `Tabs` subclass) above the single `#dialogs` `DialogListView`; `on_tabs_tab_activated` is gated by `self._started` (Tabs fires at mount, before the client exists; the flag is NOT named `_ready` — Textual's `App` already has a `_ready` coroutine) and reloads via a worker. `SidebarTabs` adds a `down`/`enter` binding that hands focus to `#dialogs` (and selects the first item) — Textual's `Tabs` only binds left/right, so without it you'd have to Tab past the strip to reach the list. The list is a `DialogListView` (a `ListView` subclass): `up` at the first item (or empty selection) jumps focus back to `#tabs` (the symmetric counterpart); anywhere else `up` scrolls the list as usual (defers to `cursor_up`). **Media send (#21):** the composer understands an `@PATH [caption]` syntax — `parse_media_command(text)` is a PURE function (no filesystem, tested standalone; `shlex` so a quoted path with spaces works, the remainder is the caption) returning `(path, caption)` or `None` (text not starting with `@`, or `@` with no path). `on_input_submitted` parses first: a media command with an existing path runs `_send_media` in a worker (path existence checked in the handler, BEFORE the worker/network — a missing file → `self.notify(severity="error")` + warning log, nothing sent); a non-media message falls through to the prior `_send_text` worker. **Live feed = TWO drain workers** started in `on_mount`: `_drain_incoming` (`listen_all()`, `out=False`, may fire a suggestion) and `_drain_outgoing` (`listen_outgoing()`, `out=True`, NO suggestion — those are incoming-only) — so our own messages sent from another device (phone/CLI/web) appear right-aligned in the open dialog. Both filter on `ev.dialog_id == self._current`. Echoes of messages sent from this composer are skipped via `_sent_ids` (bounded `OrderedDict`-as-set, written by `_remember_sent` after `send_text`/`send_media` returns its `Message`) so the optimistic bubble isn't drawn twice.

### Agent (`src/tg_messenger/agent/`)
`tg-messenger agent` auto-replies to incoming DMs — **deliberately DM-only** (it consumes `listen()` and `dialogs(dm_only=True)`; group messages never trigger it even from allowlisted users): a LangGraph router classifies each message as **chat** (single `init_chat_model` call), **task** (a `deepagents.create_deep_agent` with Telegram tools + web search) or a **custom intent** from `agent.json`, then replies via `client.send_text`. Photos go to a **vision** model (`TG_AGENT_VISION_MODEL`, falls back to the main model — which then must be multimodal); voice messages are detected and skipped with an INFO log. Configured from env: `TG_AGENT_MODEL` (`provider:model`), `TG_AGENT_ALLOWLIST` (`*` or ids/@usernames — empty is a startup error by design, `*` cannot be mixed with other entries), `TG_AGENT_SEARCH` (duckduckgo/tavily/exa/brave, lazy-imported in `search.py`), plus custom intents from `TG_AGENT_CONFIG` or `./agent.json` (see `agent.json.example`; validated fail-fast in `load_intents`, names must not collide with `RESERVED_INTENT_NAMES`). LangSmith tracing is env-only — `LANGSMITH_TRACING`/`_API_KEY`/`_PROJECT`; langchain/langgraph pick them up themselves, no graph code involved. `langsmith_tracing_enabled` (`config.py`) makes the `agent` command fail fast when tracing is on without a key and echo the status when on. **Accepted v1 risk**: the allowlist controls who can *trigger* the agent, not what it can do — an allowlisted user can have the agent read/send in ANY dialog (full-trust model, documented in `.env.example`).

- **`orchestrator.py` — `Orchestrator`**: the real LangGraph graph (route → vision | classify → chat | task | custom intents; route nodes are built from `IntentSpec`s at graph build time) with per-dialog memory (`InMemorySaver`, `thread_id` = dialog id). It never calls a model itself — `classify_fn`/`chat_fn`/`task_agent`/`vision_fn` are injected; only the deep agent's **final** answer is appended to dialog state (tool chatter never leaks). Two non-obvious invariants: images never enter checkpointed state — history keeps a text placeholder, the multimodal message goes to `vision_fn` via `_pending_images` (valid only while the runner is strictly sequential); a custom intent's `system_prompt` is prefixed to the user message **in the call payload only** (a mid-list SystemMessage breaks some providers), history keeps the original text.
- **`factory.py`** is the ONLY module importing the LLM stack (`init_chat_model`, `create_deep_agent`) — keep it that way; deepagents API drift stays contained here. `runner.py`/`tools.py`/`config.py`/`search.py`/`media.py` are stdlib+core only. The classifier prompt is generated from the intent list (`build_classify_prompt`); unknown classifier output degrades to `chat` with a warning (both in factory and, defensively, in the orchestrator).
- **`runner.py` — `AgentRunner`**: listen → skip (out=True / not in allowlist / voice / no text) → dispatch (photo → `media.download_image` → `handle(..., image=)`; text → `handle`) → reply. The allowlist check runs **before** any download; `download_image` enforces `MAX_IMAGE_BYTES` (declared size pre-download AND actual bytes). A failing message is logged (`logger.exception`) and the loop continues. No reply loops: core subscribes with `NewMessage(incoming=True)`, plus a defensive `out` check. While handling, the dialog shows a 'typing…' indicator via `client.typing()` — best-effort **by core contract** (`_SafeChatAction` logs its own failures and never raises), so callers use a bare `async with` without defensive wrappers.
- Tools in `tools.py` are plain async functions over client methods (flood-wait retry included for free); their docstrings/annotations ARE the tool schema deepagents shows the model.
- **`suggest.py` — `Suggester` (#17, `tg-messenger suggest`)**: drafts a reply in the contact's-eye style for a human to review — DELIBERATELY a draft, never an auto-reply (full automation is #18). Like the orchestrator it NEVER imports the LLM stack — `suggest_fn` is injected (`factory.make_suggest_fn`/`build_suggester` own the single `init_chat_model` import), so Suggester, the style profile and its storage are all testable on a bare `[dev]` install (no langchain, no `importorskip`). `suggest(dialog_id)` pulls history (chronological, own/peer marked via `out`), loads the per-dialog `StyleProfile` if a Storage is wired, and calls `suggest_fn(context, profile)`; a failing `suggest_fn` is `logger.exception`'d and re-raised (UIs surface the error, never silently blank). `build_style_profile` (pure) aggregates avg length / emoji freq / greetings / signatures + up to 10 example replies from OWN messages that directly follow an incoming one; empty history → a zero/empty stub. Profiles persist in `style_profiles` (SQLite #13, `register_suggest_migrations` before `connect()`); `learn(dialog_id)` is an EXPLICIT per-peer history pass — never a background scan (flood discipline). `record_last_read`/`watch_read_receipts` persist outbox read-receipts (`listen_reads`, `outbox=True`) into the kv table (`last_read_<dialog>`) — the watcher RUNS as a background task next to the `ghostwrite` and `heartbeat run` loops (created with `asyncio.create_task`, cancelled in `finally` — NOT gathered, so a Ctrl+C from the main loop never leaves it dangling); v1 only records the signal, the suggester does not act on it. Wired into the CLI (`make_suggester` seam), the web (`build_app(suggester=)`, `GET /dialogs/{id}/suggest`, 💡 button) and the TUI (incoming message in the open DM → 💡 hint, Tab accepts into the composer; gated on an empty composer so a draft is never clobbered; LLM call only in a `run_worker`).
- **`ghostwrite.py` — `GhostwriteEngine` (#18, `tg-messenger ghostwrite`)**: stage 2 of the suggester — actually auto-replies in the owner's style in EXPLICITLY enabled DMs (reuses the #17 `Suggester`, which is INJECTED — so, like #16/#17, this module NEVER imports the LLM stack and engine/storage tests need no `importorskip`). ModerationEngine pattern: `run()` fans out `listen()` (auto-reply) + `listen_outgoing()` (auto-pause) via `asyncio.gather` (NOT TaskGroup — Ctrl+C). **dry-run is the default** (`enforce=False` → "would …" log + `dry_run=1`); a reply fires only under `enforce=True`. Safeties (the heart of the issue): a **per-dialog allowlist** in SQLite (`ghostwrite_dialogs`, `register_ghostwrite_migrations`) separate from `TG_AGENT_ALLOWLIST` — **`*` is forbidden by design** (enforced at the CLI); a **per-dialog hourly rate cap** (`max_per_hour`, sliding in-memory window, bounded `OrderedDict`, injected `clock`) — exceeded → skip + warning (checked BEFORE the suggester so a maxed dialog costs no LLM call); **`pause-all`** kill switch (`paused_until` = far future); **auto-pause on human intervention** — `on_outgoing` pauses the dialog `pause_on_human_sec` (a day) when an outgoing message is NOT one the engine sent (matched against a bounded own-id cache, DeletionWatcher pattern — sentinel value is `True`, not `None`, so `pop` distinguishes a hit). Empty/whitespace drafts are skipped (never send blank); every auto-reply is journalled in `ghostwrite_log`. `is_active(storage, dialog_id, now=)` = enabled AND not paused. CLI: `ghostwrite [--enforce]` (logs enabled dialogs on start, Ctrl+C clean via `make_suggester`/`make_storage` seams) + `ghostwrite-dialogs enable/disable/list/pause-all/resume`. v1: CLI only (no web/TUI); rate window + own-id cache are in-memory (restart resets; `paused_until` persists). LLM only via factory.
- **`heartbeat.py` — `HeartbeatService` (#19, `tg-messenger heartbeat run`)**: sends scheduled "I'm still here" pings to peers by plans stored in SQLite (#13). Pydantic `HeartbeatPlan(peer, templates, interval_hours, jitter_minutes, quiet_start, quiet_end, enabled, last_ping_at, max_per_day, pings_today, day_start)` (templates non-empty, validated fail-fast); CRUD `add_plan`/`list_plans`/`list_enabled_plans`/`remove_plan` (templates JSON, PK `peer` → upsert) + a `heartbeat_log` journal register via `register_heartbeat_migrations(storage)`. **`clock` AND `rng` are injected** so a tick is deterministic and tests never sleep. The pure `is_due(plan, now=, rng=)` gates a ping by **every safety**: `interval_hours` + `jitter_minutes` (rng-chosen offset), **quiet hours** (`quiet_start`/`quiet_end` are LOCAL hours, the window wraps midnight), and **max_per_day** (`pings_today` resets when `day_start` is >24h old). The service follows the `watch.py`/Moderation pattern: `run()` is a thin once-a-minute loop fanned out via `asyncio.gather` (NOT TaskGroup — Ctrl+C); the single tick is the directly-testable `process_plans(now=)`. Per due plan it also enforces the **«наше последнее сообщение без ответа»** safety (reads `client.history(peer, limit=1)`; if the tail is OURS, `out=True`, the собеседник hasn't replied → skip + INFO log), then picks a template via `rng` OR, if an injected `text_provider` (the суфлёр hook — a plain `Callable`, NO LLM import) is given, its text (its failure or empty result → warning + fallback to a template), sends via `client.send_text`, updates `last_ping_at`/`pings_today`/`day_start`, and journals. A send error is `logger.exception`'d and the loop survives. CLI: `heartbeat plan PEER (--at HH:MM | --interval H) --template … [--jitter --quiet-start --quiet-end --max-per-day]` — **`--at` is a one-shot native server-side scheduled send** (`send_text(schedule=)`, no stored plan), **`--interval` stores a recurring plan**; plus `heartbeat list/remove PEER` and `heartbeat run` (Ctrl+C clean via the `make_storage` seam). **TZ-допущение**: `now` is a Unix wall-clock timestamp; the quiet-hours check maps it to a local hour via `datetime.fromtimestamp(now)`. v1: CLI only; the daily counter / scheduling state persist in SQLite. **`client.send_text(peer, text, reply_to=None, schedule=timedelta|datetime|None)`** — `schedule` defers the send server-side (Telethon `send_message(schedule=)`); default `None` keeps the prior behaviour.

### Interop (`src/tg_messenger/interop/`) — #20
The ONLY package that talks HTTP to [tg_content_factory](https://github.com/axisrow/tg_content_factory) (httpx, optional `[interop]` extra). **`core/` never imports httpx** (packaging test enforces it); `agent/tools.py` imports `interop` LAZILY inside the factory tool bodies, so a bare install stays httpx-free. Role split: **messenger = hands** (reads/sends via the account), **factory = memory + search** (archive index + task queue).
- **`factory_client.py` — `FactoryClient(base_url, password, *, http=, sleep=)`**: async httpx wrapper. `http` (an `AsyncClient`) and `sleep` are injectable — tests pass `httpx.MockTransport` (respx is NOT installed) + a no-op sleep, so retries cost no real time. **Auth = HTTP Basic with an EMPTY username + the shared password** (`httpx.BasicAuth("", password)`) — a mock contract the factory matches in its companion issue; documented choice. `_request` maps failures: a 401 → `FactoryError` with a clear message (never a bare httpx exception); transient `httpx.TransportError` (connect/read/network) → up to `MAX_ATTEMPTS=3` retries with linear backoff through the injected sleep; any 4xx/5xx response is NOT a network error → straight up as `FactoryError`, no retry. Methods: `search_messages(identifier, query, limit, date_from, date_to, topic_id)` (GET `/search/messages/{identifier}`), `create_task(type, payload)` (stamps `payload["v"]=PAYLOAD_VERSION` for versioning), `get_task`, `claim_next(types)` (404/empty → `None`), `complete_task(id, result_payload)`, `fail_task(id, error)`. `InteropTask` is the Pydantic model (`id/type/payload/status/result_payload`).
- **`worker.py` — `Worker(client, factory, *, types, agent=None, sleep=, idle_sleep=)`** (`tg-messenger worker`): the `AgentRunner` loop shape — poll → claim → execute → complete|fail, where one failing task NEVER kills the loop (`logger.exception` + `fail_task`, loop survives). `process_once()` is the single directly-testable step (`run()` is the production loop with an idle sleep on an empty queue). Executors by type: `dm_reply`/`chat_answer` (`{peer, text}` → `client.send_text` → `{sent: msg_id}`; with `{peer, prompt}` instead of `text` it needs the injected `agent` — without it the task fails with a clear "requires the [agent] extra" message), `fetch_history`/`fetch_dialogs` (read → `model_dump(mode="json")` → `{messages|dialogs: [...]}`). The agent is OPTIONAL and injected (the langchain-free worker tests need no `importorskip`; only the prompt-path test does).
- **Agent factory tools (`agent/tools.py`)**: `make_telegram_tools(client, *, factory_url=, factory_password=)` appends `factory_search`/`factory_create_task` ONLY when `factory_url` is set (from `TG_FACTORY_URL`/`TG_FACTORY_PASSWORD` on `AgentConfig`). They build the `FactoryClient` via the `_make_factory_client` seam (tests patch it — no real httpx) and import `interop` lazily inside that seam.
- **CLI/config**: `tg-messenger worker --factory-url URL [--types …] [--interval N]` — `--factory-url` required (or `TG_FACTORY_URL`); lazy `from tg_messenger.interop.worker import Worker` → `ClickException "pip install 'tg-messenger[interop]'"` when the extra is absent; Ctrl+C clean. The command builds the optional prompt-task agent via **`make_worker_agent(client)`**: best-effort — no `[agent]` extra or no `TG_AGENT_MODEL` → `None` with an INFO log (prompt tasks then fail with the clear hint, fetch/dm_reply keep working; `require_allowlist=False` — task access is the factory's concern); the command output says `prompt tasks on/off`. Seams: `make_factory_client(base_url=, password=)` (also lazy-imports interop), `make_worker_agent` and the patched `Worker`. **Accepted v1 risk (full trust)**: factory tasks run on the account with no source auth beyond the shared password — whoever enqueues drives the hands (documented in `.env.example`).

### Test seams (how the network is faked)
Tests inject a fake Telethon client — never hit the network. Mirror these patterns when adding tests:
- **Core**: pass `client_factory=lambda session, api_id, api_hash: fake_client` to `StandaloneTelegramClient`.
- **Web**: `build_app(client=stub)`.
- **CLI**: `monkeypatch.setattr(cli_main, "make_client", lambda **kw: stub)`.
- **TUI**: `MessengerTUI(client=stub)`.
- **Agent**: `Orchestrator(classify_fn=fake, chat_fn=fake, task_agent=stub_with_ainvoke)`; factory tests monkeypatch `factory.init_chat_model`/`factory.create_deep_agent`; CLI patches `cli_main.make_agent_runner`. Agent test modules guard with `pytest.importorskip("langgraph"/"deepagents")` — the suite stays green on a plain `.[dev]` install (they skip). Tests never call a real LLM.
- **Interop**: HTTP is faked with `httpx.MockTransport(handler)` wrapped in an `AsyncClient` and injected into `FactoryClient(http=, sleep=_noop)` (respx is NOT installed; the no-op sleep makes retry tests instant). The worker is tested against an in-memory `StubFactory`/`StubCoreClient` (no httpx, no network); the agent prompt-path test guards with `pytest.importorskip("langgraph")` while every other interop test is langchain-free. CLI: `monkeypatch.setattr(cli_main, "make_factory_client", …)` + patch `tg_messenger.interop.worker.Worker`.
- `tests/conftest.py` provides `FakeTelethonClient` (records `sent`/`downloads`, counts network reads via `iter_dialogs_calls`/`iter_messages_calls` for the TTL-cache tests, can `push_event` into registered handlers — dispatched by builder type like real Telethon: `MessageDeleted` by the `deleted_ids` attribute, `NewMessage(incoming=)/(outgoing=)` by `event.message.out`) and a `session_dir` tmp fixture. Cache tests inject a fake clock (`t = {"now": 0.0}`, `clock=lambda: t["now"]`) — time never really passes — and exercise concurrency with `asyncio.gather` + `asyncio.Event`, never a real `sleep`.
- `tests/__init__.py` is deliberately empty — don't delete it. Without it `tests/` is a namespace package, and a stray top-level `tests` package in site-packages (ultralytics and yfinance_cache ship one) wins the import resolution, breaking `from tests.conftest import ...` at collection time.
